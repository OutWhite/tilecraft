# B200 Tuning Notes

## Hardware Signals

For this project, assume the target is NVIDIA B200 / Blackwell rather than a generic CUDA GPU.

Useful constraints from NVIDIA's Blackwell tuning guide:

- B200 is compute capability 10.0.
- Maximum resident warps per SM is 64 for compute capability 10.0.
- Register file size is 64K 32-bit registers per SM.
- Maximum resident thread blocks per SM is 32.
- B200 supports high-bandwidth HBM3/HBM3e memory.
- B200 has a unified L1/texture/shared-memory cache and a large L2 cache.
- Global and distributed-shared-memory accesses should be coalesced and aligned to 32-byte segments.

For vecadd, the kernel is streaming and memory-bandwidth bound:

```text
bytes = N * (sizeof(A) + sizeof(B) + sizeof(C))
      = 25,000,000 * 12
      = 300 MB
```

The theoretical B200 HBM bandwidth is several TB/s, so launch overhead, memory coalescing, vectorization, and enough resident CTAs dominate the practical score.

## VecAdd Strategy

Triton:

- Use a 1D program grid.
- Use `BLOCK_SIZE = 1024` as the first performance baseline.
- Use `num_warps = 4` so each program has 128 lanes worth of warp execution, matching a common memory kernel shape.
- Use `.cg` streaming loads and `evict_first` because A/B have no reuse.
- Use `.wb` stores for C.

CuTe DSL:

- Use 128 threads per CTA as a first baseline.
- Use 128-bit per-thread copy atoms, so `float32` maps to 4 values per thread.
- One CTA covers 512 float elements.
- Use `zipped_divide` + identity-coordinate tensor + `elem_less` for tail predicates.

## Variants To Benchmark

Vecadd is simple enough that we should keep a small variant matrix and let leetgpu timings decide:

| Variant | Threads / Program | Elements / Unit | Notes |
| --- | ---: | ---: | --- |
| Triton `BLOCK=1024, warps=4` | 128 lanes | 1024 | Current baseline |
| Triton `BLOCK=2048, warps=8` | 256 lanes | 2048 | Fewer CTAs, more work per program |
| Triton `BLOCK=256, warps=4` | 128 lanes | 256 | More CTAs, may help scheduling |
| CuTe `128 threads, 128-bit` | 128 threads | 512 | Current baseline |
| CuTe `256 threads, 128-bit` | 256 threads | 1024 | Compare launch/occupancy behavior |

For this first problem, performance should be reported as both runtime and effective bandwidth:

```text
GB/s = 300 / runtime_ms
```

because the fixed benchmark uses 300 MB of global-memory traffic.
